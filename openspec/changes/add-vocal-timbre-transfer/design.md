## Context

这是一个全新项目，目标是构建"歌曲换音色"工具。核心技术链路由四个阶段组成：

1. **源分离**：把原唱歌曲拆成"纯人声"和"伴奏"两轨。
2. **目标音色建模**：从歌手 B 的参考歌曲中提取干净人声，学习其音色。
3. **音色迁移**：把歌手 A 的人声转换为歌手 B 的音色，保留旋律/歌词/节奏。
4. **混音合成**：把转换后的人声与原曲伴奏合成最终歌曲。

当前仓库为空，需要从零搭建 Python 项目骨架。人声音色迁移是计算密集型任务，业界已有成熟的开源方案（Demucs 做源分离、RVC/So-VITS-SVC 做歌唱声转换）。本设计聚焦于如何把这些组件编排成一条易用、本地优先、可复现的流水线。

约束：
- 用户音频仅在本地处理，不上传云端（隐私优先）。
- GPU 优先，但必须提供 CPU 回退。
- 首版面向单机命令行使用，后续可扩展 Web UI。

## Goals / Non-Goals

**Goals:**
- 提供一条端到端命令：输入原唱歌曲 + 目标歌手参考歌曲 → 输出换音色歌曲。
- 模块化设计，每个阶段（分离/建模/迁移/合成）可独立调用与替换。
- 本地优先，支持 GPU 加速与 CPU 回退。
- 支持多首参考歌曲融合目标音色，提升音色稳定性。
- CLI 优先，预留 Web 接口扩展点。
- 可配置输出格式、采样率、转换强度等参数。

**Non-Goals:**
- 不做实时换音色（首版仅离线批处理）。
- 不做歌词/旋律的修改或重新生成（仅替换音色，保留原曲内容）。
- 不做云端 SaaS 或多用户服务（首版单机）。
- 不训练自有基础模型，首版基于现有开源预训练模型集成。
- 不做专业级混音母带处理（仅做基础对齐与合成）。

## Decisions

### 决策 1：源分离采用 Demucs v4 (htdemucs)
- **选择**：Demucs（Meta 开源，MIT 许可）。
- **理由**：在 MUSDB18 等基准上分离质量领先，支持 2 轨（人声/伴奏）与 4 轨分离，PyTorch 原生、社区活跃。
- **备选**：Spleeter（TensorFlow，速度快但质量略低、维护趋缓）。首版优先质量。
- **影响**：分离质量直接决定后续音色迁移的干净度。

### 决策 2：音色迁移引擎采用 RVC（Retrieval-based Voice Conversion）
- **选择**：RVC 作为主力引擎。
- **理由**：基于检索的特征索引方式，推理轻量、训练/微调快，对歌唱场景支持良好，社区与预训练资源丰富，支持 pitch 算法（crepe/rmvpe）保持音高。
- **备选**：So-VITS-SVC（专为歌唱设计，质量高但训练更重、推理更慢）。设计上保留 `VoiceConverter` 抽象接口，未来可插入 So-VITS-SVC 后端。
- **影响**：需要为目标歌手构建 RVC 特征索引（.index）及可选微调模型。

### 决策 3：参考音色建模 = 人声提取 + 特征索引构建
- **选择**：先用 Demucs 从参考歌曲中提取歌手 B 的干净人声，再用 RVC 的内容编码器构建检索特征索引；多首参考的特征做融合。
- **理由**：检索式方法无需完整重训，多首参考能提升音色覆盖度与稳定性；比完整微调门槛低、速度快。
- **备选**：对目标歌手做完整 RVC 微调（质量更高但耗时长、需要更多数据）。预留为"高质量模式"可选项。

### 决策 4：技术栈 Python + PyTorch + Typer(CLI) + FastAPI(Web 预留)
- **选择**：Python 3.10+，PyTorch（CUDA/MPS/CPU），Typer 做 CLI，FastAPI 预留 Web 层，librosa/soundfile/pydub 做音频 I/O。
- **理由**：音频 ML 生态以 Python 为主；Typer 提供类型友好的 CLI；FastAPI 便于后续加 Web UI 而不重写核心。

### 决策 5：模块化流水线 + 插件式后端
- **选择**：定义清晰阶段接口（`SourceSeparator`、`VoiceModeler`、`VoiceConverter`、`Remixer`），每阶段有默认实现但可替换。
- **理由**：降低耦合，便于单独优化或替换某个阶段（如换用 So-VITS-SVC、换混音器），也便于单元测试。

### 决策 6：模型本地缓存与懒加载
- **选择**：预训练模型首次使用时下载到本地缓存目录（`~/.musicremix/models`），后续直接复用；运行时按需加载。
- **理由**：模型体积大，避免重复下载与打包进仓库。

## Risks / Trade-offs

- **[GPU 依赖]** 音色迁移在 CPU 上极慢（一首歌可能数分钟到数十分钟） → 提供自动设备检测，CPU 回退时给出明确耗时提示与降采样选项。
- **[分离残留]** 源分离可能残留伴奏/和声，污染音色迁移输入 → 在迁移前可选人声清洗/高通滤波；暴露分离质量指标。
- **[音色漂移/电音感]** 转换强度过高会产生机械感 → 暴露 `index_rate`/`f0` 等参数，提供默认安全值与"自然度优先"预设。
- **[参考歌曲质量不一]** 参考含伴奏/噪声会降低音色模型质量 → 参考歌曲同样经过源分离，仅用干净人声建模。
- **[模型许可证]** RVC/So-VITS-SVC 部分模型/代码许可需逐一核查 → 文档化各依赖许可证，商用前复核。
- **[对齐漂移]** 转换后人声与伴奏可能轻微时长不对齐 → 合成前做基于伴奏的时长大校准/重采样对齐。
- **[内存峰值]** 长歌曲处理时显存/内存峰值高 → 按片段（chunk）流式处理并拼接。

## Open Questions

- 是否首版就引入 So-VITS-SVC 微调作为"高质量模式"，还是仅做 RVC 检索式迁移？（建议首版仅 RVC，微调作为后续增强）
- 是否需要自动音域匹配（把源人声 pitch 调整到目标歌手舒适音域）？（建议作为可选参数，默认关闭）
- Web UI 是首版必要还是后续迭代？（建议后续迭代，首版 CLI 优先）
- 是否支持目标音色为"多歌手混合"的创意玩法？（建议作为 pipeline 可选输入，非首版重点）
